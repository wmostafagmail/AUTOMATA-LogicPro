# Architectural Design Notes (ALU\_E2E)

## 1. Data Flow and Datapath
The datapath is straightforward, consisting of two primary inputs (A, B) feeding into a multiplexer structure controlled by the `OpCode`. The selected operation then computes the raw result. All calculations use 8-bit widths derived from `ieee.numeric_std` for safety.

## 2. Control Unit
The control logic is implemented using a large combinational block (`case` statement) that maps the 3-bit `OpCode` input to select the appropriate operation path: Add, Sub, Logic Gate (AND/OR/XOR), or Shift.

*   **Shift Operation:** The shift amount is assumed to be defined by an implied constant or derived from the OpCode structure for simplicity in this fixed-width ALU model, focusing on functional correctness rather than flexible shift amounts.
*   **Flag Generation:** Flag calculation (Zero, Carry, Overflow) happens *after* the primary result computation and utilizes dedicated arithmetic logic within the architecture to ensure dependency correctness.

## 3. Flag Calculation Details
The status flags are derived combinatorially from the calculated internal result (`Result_INT`) and the operation performed:

*   **Zero Flag (Z):** $Z = \text{NOT} (\text{Result\_INT} = '0')$. True if all bits of `Result` are zero.
*   **Carry Flag (C):** Only calculated for ADD/SUB. The carry-out bit generated by the 8-bit arithmetic unit is captured.
*   **Overflow Flag (V):** Only calculated for ADD/SUB using signed comparison. Overflow occurs if $A_{sign} \ne B_{sign}$ and $(\text{Result}_{sign} \ne A_{sign})$ or $(\text{Result}_{sign} \ne B_{sign})$. More simply, it detects if the sign of the result differs from the expected combined sign based on the inputs' signs (requires conversion to `signed` type).

## 4. Design Principles
The implementation strictly adheres to synthesizability rules: all logic is combinational or synchronous; no inferred latches are used; and signal assignments rely only on defined input/internal signals, ensuring clean state transitions.
